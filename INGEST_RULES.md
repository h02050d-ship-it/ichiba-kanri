## 0. 前提・環境（Windows実機で確認済み）

- **PDF→PNGはWinRT `Windows.Data.Pdf`で可能**（poppler/node不要）。動作スクリプト＝本フォルダ `convert_all.ps1`（幅1700〜2000pxでAI読取に十分）。
  - PS5.1では `[System.WindowsRuntimeSystemExtensions]` 型アクセラレータが解決不能。**GACのDLLを `[Reflection.Assembly]::LoadFrom(...System.Runtime.WindowsRuntime.dll)` → `$asm.GetType("System.WindowsRuntimeSystemExtensions")` で取得**して AsTask を呼ぶ（本フォルダのスクリプト参照）。
  - `$page.Close()` は存在しないメソッドだが try/catch で無害化。`$stream.Flush()` も存在しないので使わない（Disposeでフラッシュされる）。
  - **⚠️ .ps1に日本語コメント/文字列を書くとPS5.1がANSIで誤読して壊れる**。スクリプト本体はASCIIのみ。日本語パス/シート名は UTF-8テキストファイルに書いて `Get-Content -Encoding UTF8` で読む（`xlsx_path.txt` 方式）。
  - 巨大な一行コマンドは `EPERM uv_spawn` になりやすい → スクリプトファイル化して `& script.ps1` で実行。
- **PowerShellは `Invoke-RestMethod` を使う（curlは302で壊れる）**。
- Google Drive上のファイルは**読むだけ**。移動・変更・削除しない。
- **push先GAS exec URL（確定・URL不変運用）**: `https://script.google.com/macros/s/AKfycbzdr8jQQKW1dqqGKd1ZtIgUgrlLgoe53-lyXdrIkCAORI23-BMU_6pWy9WyZZaHKQ5Tjw/exec`
- **PUSH_SECRET は `C:\Users\hayaz\ichiba-kanri\.secret` を読む**（gitignore対象・設定済み）。
- PNG化スクリプトは本リポジトリ `tools\convert_all.ps1`（`pdf_list.txt`方式）/ 投入の実例は `tools\push_initial.mjs`（node実行可・fetch redirect:follow）。
- **pushSnapshot には必ず `docTotals: {bundles, volume}`（書類の合計欄の値）を付ける**。GASがrowsから読取合計を自前計算し、書類⇔読取の突合判定（checksum_ok）と ingest_log への両値保存を行う＝Webツール「⚙ 設定･検証」タブで人が比較確認できる。

## 0.5 精算書ライブラリへのコピー（毎日・必須）

`G:\マイドライブ\グーグルドライブ\02_経理\01.会計帳簿\市場売上明細\精算書ライブラリ\` に市場別サブフォルダ（勝山/金平/千葉県協組/HIKARI/中央木材/吉貞佐野/吉貞熊谷/丸宇/アイザワ/ナイス各拠点/99_要確認）がある（2026-07-21新設・既存394件集約済み）。Webツールの📁ボタンはここを指す。
- 監視フォルダで見つけた**新しい市場帳票PDFは、取込対象かどうかに関わらず必ず該当市場フォルダへコピー**する（Copy-Itemのみ・元は移動しない・同名スキップで冪等）。
- 文字化け名はコピー側だけ `YYYYMMDD_市場名_帳票名.pdf` に直してよい。
- 市場判定できないものは `99_要確認\` へ。無関係書類（領収書・法務書類等）はコピーしない。
- これによりアーカイブ整理（96時間で月別へ移動）後もライブラリで常に見つかる。

## 1. 共通フロー

1. **監視フォルダを走査**して前回処理以降の新規PDFを検出（下記フォルダ）。
2. **文字化けファイル名対策**: ワイルドカードで拾う（`*不落*` `*元落*` `*預材*` `*精算*` `*ナイス*` 等）。ファイル名先頭の `YYYYMMDD_` を snapshotDate の第一手掛かりにする（ただし**帳票内の日付が正**＝金平は名前2026-07-13でも中身は06-26市）。
3. **PDF→PNG**: `convert_all.ps1`（`pdf_list.txt` に `パス|プレフィックス|幅` で列挙 → 実行）。出力は作業フォルダ配下のpngへ。
4. **AI読取**: 市場別レシピ（§3）でPNGを見て明細抽出。
5. **正規化**: `dict` シート（品名/等級/巾/市場名/除外）で統一表記へ。長さm→mm・巾cm/働き巾→呼び巾120・厚cm→mm。
6. **検算**: 書類の合計欄（【明細計】【元落合計】【旧元落計】【合計】【預材 合計】材積）と、抽出明細の**束数合計・材積合計が一致**するか。
   - 一致 → `checksumOk=true, status=ok`。
   - 不一致 → **`status=warn`・message に不確実点を書いて push**（握りつぶさない。人間が後で `confirmed` チェック）。
7. **push**: `pushSnapshot`（PUSH_SECRET照合）でGASへ。市場ごとに**全置換**（元落/不落/預材＝その市場の全在庫のため）。冪等: 同 batchId 再送で二重計上なし。

## 3. 市場別レシピ（v1で在庫が直接分かる市場のみ push 対象）

### 3-1. 勝山（（株）勝山木材市場）— 不落明細書
- フォルダ: `市場売上明細\勝山売上\` パターン `*勝山*明細*`。最新1枚のみ（全在庫が載る）。
- 帳票: **不落明細書**。ヘッダに `令和8年07月15日` `3403回市`。列＝桟番号/置場/樹種/等級/品名/欠点/長さ/厚/幅/入数/**数量**/**材積**/単位/単価/金額/区分/摘要。
- 読む: `***不落明細***` 以下の全行。**bundles=数量列**。
- 正規化: 品名は全行空欄→**桧フローリング**（厚1.5＝15mm）。等級空欄→**丸節**・`ム`→無節・`上小`→特上小・`小節`→小節・`丸節`→丸節。幅`10.8`（働き巾）→**呼び巾120**。長さ`4.0/3.0/2.0`m→4000/3000/2000。
- **入荷市日なし → stale_basis=初出近似**（ingestMarketDate=null）。
- 検算: 最下部 `明細計` の 数量合計 と 材積合計。実測=**506束 / 21.0504㎥**（内訳 丸節282/特上小120/無節72/小節32）。
- snapshotDate = 帳票の市日（`令和8年07月15日`＝2026-07-15）。batchId 例 `katsuyama_20260715`。

### 3-2. 金平（（株）金平）— 売上精算書並びに元落通知書
- フォルダ: `スキャンスナップ\アップロード済\` パターン `*元落*`（**ファイル名先頭が文字化け**）。宇都宮営業所。
- 帳票: 上部に売れた明細（単価/金額あり）→`【元売上計】【売上合計】`。**その下＝元落（単価・金額が空の行）＝在庫**。列＝入荷市日/市回数/林場・号数/伝票番号/材種/**等級**/長/厚/巾/入数/**個数**/**材積**/単価/金額/備考/買番。
- 読む: `【売上合計】` より下の全行（単価/金額空）。**bundles=個数列**。
- **⚠️ 等級列に品名が混在**: `本実目透 羽目板`→**桧羽目板(厚1.2=12mm)**・`特上小フローリング`→桧フローリング特上小・`本実上小`→桧フローリング特上小。品名記載なし→桧フローリング(厚1.5)。
- 等級正規化: `上小`→特上小・`ムジ`→無節・`丸節/無節/特上小`はそのまま。`※`印＝旧元落マーカー（等級語から除去）。
- **入荷市日=和暦YYMMDD → 西暦変換**して ingestMarketDate に、原文（例 `240107`）を ingestMarketRaw に保持:
  - YY=先頭2桁。`1<=YY<=8`→令和→西暦=`2018+YY`。`20<=YY<=31`→平成→西暦=`1988+YY`。
  - 例: `240107`→平成24→2012-01-07（最古・14年寝ている）/ `080306`→令和8→2026-03-06 / `280826`→平成28→2016-08-26。
  - `09<=YY<=19` は空白帯（未使用）。範囲外は null→初出近似へ。
- **stale_basis=入荷市日**。
- 検算: 末尾 `【旧元落計】` `【元落合計】` の 個数 と 材積。実測=**200束 / 10.5433㎥**（全て旧元落）。
- 注意: ファイル名の日付（例2026-07-13）と帳票の市日（06-26）が違う。**snapshotDate=帳票の市日**（2026-06-26）。
- 注意: 240107行は入数欄が空だが材積0.0432/6束＝入数1と整合（入数=1で記録・要人間確認）。本実目透 羽目板は等級表記なし（gradeブランク・要確認）。

### 3-4. ナイス沼津・小牧 — ご請求書(契約)兼預材明細表（帳票ID SHL01101）
- フォルダ: `市場売上明細\ナイス売上\沼津\` `\小牧\` パターン `*ナイス*沼津*` `*ナイス*小牧*`。左上に **`SHL01101`**＝預材あり。
- 帳票: 上部＝売れた契約明細（単価/金額あり）→`【合計】`。**下部 `《預材一覧》`＝預けている在庫**。列＝メーカ名/商品名/商品コード/入荷日/伝票No/倉庫/置場/サイズ/**束入**/**BL入**/**数量**/**材積**/単価/金額/備考。
- 読む: `《預材一覧》` の全行（`【預材 合計】` まで）。
- **⚠️⚠️ 束数 = 数量 × BL入**（数量=口数、BL入=1口あたり束数）。**`【預材 合計】` の個数（=行数/口数）を束数と誤読しない**。実例: 沼津 数量1×BL入3=3束、数量1×BL入18=18束＝計21束（合計欄の"2"は口数）。小牧 数量1×BL入22=22束。

- 注意: **小牧のPDFは90度回転していることがある**（縦組みで出力）。PNGを見て向きを判断し読む。
- snapshotDate = `◯◯年◯◯月◯◯日締`（対象期間の締日。例 2026年06月20日締＝2026-06-20）。

## 5. pushSnapshot body 仕様（設計書§2-6 転記）

エンドポイント: GAS exec の POST。body は **base64(UTF-8) JSON**（日本語化け対策）。

```
{ "action":"pushSnapshot",
  "secret":"<PUSH_SECRET>",                 // = C:\Users\hayaz\ichiba-kanri\.secret を読む
  "market":"（株）勝山木材市場",              // markets.market と完全一致（正規化辞書で寄せる）
  "snapshotDate":"2026-07-15",              // 帳票の市日 yyyy-MM-dd
  "batchId":"katsuyama_20260715",           // push単位ID（全置換の対象特定）
  "srcPdf":"20260715_勝山_明細書の送付.pdf", // ファイル名 or Driveリンク
  "checksumOk":true,                        // 書類合計欄と明細合計が一致
  "status":"ok",                            // ok | warn（検算NGでも⚠️で受ける）
  "message":"検算一致 506束/21.0504m3",      // 不確実点・警告文
  "rows":[
    { "product":"桧フローリング", "grade":"丸節",
      "thickness":15, "width":120, "length":4000, "irikazu":8,
      "bundles":27, "volume":1.3986,
      "ingestMarketDate":"2025-04-17"|null,  // 読めた市場のみ。金平=和暦変換後/千葉沼津小牧=西暦短縮変換後
      "ingestMarketRaw":"070417",            // 原文（監査用）
      "note":"に6/桟1427" },
    ...
  ] }
```

- 認証: `body.secret === ScriptProperties.PUSH_SECRET`。不一致→`{ok:false,error:'forbidden'}`。ユーザーpass照合は不要。
- 動作（stock_method=直接）: LockService排他 → 当該market の既存stock全削除 → rows挿入（first_seen_dateは旧値引継ぎ）→ markets.latest_snap_date/bundles 更新 → ingest_log 追記。
- 動作（積み上げ/要確認）: v1は全置換しない（受けても無視 or 手動運用）。
- **冪等**: 同 batchId 再送で同結果。

## 6. 失敗時運用

- AI読取が曖昧/検算NG → `status=warn`・message に不確実点を書いて push（在庫は更新しつつ⚠️で浮かせる）。ingest_log に残る。
- UIサマリー/詳細で⚠️表示 → 人間が設定画面 or スプシで `confirmed` をチェック。
- 数字に自信がなければ**勝手に合わせない**。原文（ingestMarketRaw等）を残し message に明記。

## 7. 売上明細の取込（sales_lines・棚卸しアンカー方式の販売側）

在庫が直接取れない市場（丸宇4拠点/HIKARI/中央木材西部・大口/アイザワ）は
**推定在庫 = 棚卸しアンカー（anchors・年1回リセット） ＋ 搬入（納品書DB自動参照） − 販売（sales_lines）** で出す。
搬入はGASが納品書DB（取引方法=市売）を自動集計するので取込不要。**取込が必要なのは販売側=精算書の売上明細のみ**。

### 対象市場と読取箇所
| 市場 | 帳票 | 売上明細の読み方 |
|---|---|---|
| 丸宇（北浜/下館/京葉/大栄浜） | 計算書 | 売上明細行（OCR文字層なし＝PNG化して画像読取）。規格×束数×単価 |
| （株）HIKARI | 市売精算書 | 売上明細行（規格×束数×単価×金額）。**⚠️「新入荷売上」も必ず計上する**（当日入荷・当日販売分。搬入側=納品書DBに出荷が載るため、売上から除外すると推定在庫が幽霊化する＝2026-07-21に180束分の実害を修正済み）。除外するのは「元落（単価/金額なしの未売れ行）」のみ。docTotals=売上合計（新入荷売上計＋元落売上計） |
| 中央木材（西部/大口営業所） | 荷主精算明細書 | **3枚目**の明細（特=丸節等はdictで正規化） |
| (株)アイザワ | 精算書（明細なし・金額のみ） | 明細が無い→**材積総数 ÷ 0.0576㎥ で束数換算**（15mm×4000×120mm×8枚=0.0576㎥/束）。⚠️単一規格前提の近似（実際は12mm/羽目板/2mも少量あり＝2026-07-21納品書DB実測。主力15×4m丸節が67%。換算値はwarn付きでpush） |

### pushSalesLines 仕様（GAS v4・PUSH_SECRET認証）
```
POST <exec>  body = base64(UTF-8 JSON)
{ "action":"pushSalesLines", "secret":"<PUSH_SECRET>",
  "market":"（株）HIKARI",            // markets.market の正本名
  "settleDate":"2026-07-15",          // 精算書の日付
  "batchId":"hikari_sales_20260715",  // 冪等キー（同一batch_idは削除→挿入）
  "srcPdf":"ファイル名 or Driveリンク",
  "status":"ok|warn", "message":"検算一致 / 警告文",
  "docTotals":{"bundles":123,"volume":5.1234},   // 書類の合計欄（あれば必ず付ける→GASが検算）
  "rows":[ { "product":"桧フローリング","grade":"丸節",
             "thickness":15,"width":120,"length":4000,"irikazu":8,
             "qty":10,               // 売れた束数（正数）
             "unitPrice":7700,"amount":77000,"volume":0.576 }, ... ] }
```
- 同一 batchId 再送は削除→挿入で冪等（二重計上しない）。
- ingest_log に **kind=sales** で記録（在庫スナップの kind=snapshot と区別）。
- docTotals があれば GAS が rows合計と突合して checksum_ok を自動判定（束数完全一致かつ材積差<=0.01）。

### アンカー（anchors）の運用
- 年1回の棚卸しで `saveAnchor`（ユーザー認証API or Webツール設定画面）に market×規格×束数×anchor_date を登録。
- アンカー日以降の搬入・販売だけが推定計算に入る。棚卸しをやり直したら anchor_date を新しくする＝自動リセット。
- 対象市場は markets.stock_method=`推定` に設定（saveMarketで変更可）。

### ⚠️ 納品書DBの取引先表記ゆれ（2026-07-21実測・dict scope=市場名 への投入必須）
納品書DB「データベース」の取引先列は同一市場でも複数表記が混在。推定計算はdict正規化＋空白無視で突合するが、以下は dict に登録しておく:
- 丸宇: 「丸宇木材市売（株）北浜市場」「丸宇木材市売（株）　北浜」「丸宇木材市場（株）　北浜」等（市売/市場・全角空白・「市場」有無の揺れ）。**4拠点目=大栄浜市場と判明**（「丸宇木材市場（株）　大栄浜市場」等3表記・計809束）。
- 中央木材: 「中央木材市場（株）」と「中央木材市売（株）」×「西部営業所/大口営業所」の4表記。
- アイザワ: 納品書DBは「(株)アイザワ」（半角カッコ）。

## 8. 棚卸明細の取込（tanaoroshi・半期/決算＋随時）

市場から届く棚卸明細（例: 金平「委託材荷主別在庫明細表」）をAI読取→pushTanaoroshi→GASが理論在庫と突合。Webツール「⚙設定･検証＞📋棚卸チェック」に申告⇔理論の差異が出る。

- **監視フォルダ（正式な定位置）**: `G:\マイドライブ\グーグルドライブ\01_FAX・書類\棚卸関連\<期>\`（Drive ID 1mhHew_SE07KA3nJxorWXaiiq_PHtD0rb。期フォルダ例: 20260331 / 202509 / 202503）。FAX受信で自動的にここへ入る運用。新しい期フォルダ・新ファイルを検知したら取込。
- ファイル名: `YYYYMMDD_市場名.pdf`（受信日）。**asofDate=書類内の指定日（期末日）が正**（例: 指定日〜8/03/31＝2026-03-31）。
- 読み方: 荷主=林材木店 の行のみ。列は市場様式による（金平: 入荷市日/状場・号数/伝票NO/材種/等級/長/厚/巾/入数/個数/材積。※印=旧元落）。等級に品名が混ざる場合は §3/§7 と同じ正規化。入荷市日があれば ingestMarketDate に（和暦/西暦短縮の変換規則は §3-2/§3-3）。
- 検算: 《荷主計》《総合計》の伝票枚数・個数・材積と突合。docTotals必須。
- push: `pushTanaoroshi` {market, asofDate, batchId(例 tana_kanehira_20260331), srcPdf, docTotals, rows}。同一batchId置換で冪等。
- 突合後の運用: 差異はWebツールで人間が確認。**推定市場は確認後「この棚卸をアンカーに採用」ボタンでリセット**（adoptAnchor）。直接市場は差異＝読取ミスか記録漏れのシグナル。
- ⚠幽霊在庫の教訓（2026-07-22金平）: 市場の台帳に明細だけ生き続けている在庫があり得る。滞留年数が異常に古い申告行は「実体なし」の可能性を疑い、ユーザーに確認する。
